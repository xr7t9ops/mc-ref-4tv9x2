1HBB Clearance Check — Session-Start Environment Audit (v3.4 — landed via claude.ai Opus 1CPB)

Every node MUST verify at session start: is the 1HB on the right platform, interface, model, device, and settings for this task? If not, redirect before doing work. This is not optional.

The last B in 1HBB is the bottleneck. The AI must actively reduce it — not just work within it. If Paul is on claude.ai browser but the task needs Claude Code file ops — say so, redirect, and wait. If Paul is on Haiku but the task needs Opus-level reasoning — say so, redirect, and wait. Burning tokens on the wrong surface is a 1HBB violation. Silently doing wrong-fit work is the AI accepting the bottleneck instead of clearing it.

Clearance Checklist — verify before Step 1 of any task:

  • Platform — Claude Code / claude.ai / Cowork / Desktop / Mobile. Is this surface right for the task?
  • Interface — CLI / IDE / Browser / App. Does the task need features only one of these offers (file ops, canvas, MCP servers, etc.)?
  • Model — Opus 4.6 (deep reasoning, architecture) / Sonnet 4.6 (balanced execution) / Haiku 4.5 (fast/cheap routing). Is the model-to-task fit correct, and is Extended Thinking on when it matters?
  • Device — Windows desktop with Drive mount / laptop / phone. Can this device physically complete the task (file system access, port mapping, local repos)?
  • Settings — Required tools enabled? Required skills installed? MCP servers connected? Correct account logged in? ENOC clearances granted?

If any clearance item fails, halt, report the mismatch, and propose the correct surface. The 1HB decides whether to switch or to proceed with the mismatch acknowledged. The AI may not silently absorb the friction — that is how the bottleneck compounds across sessions.

Micro-Node Swarms — Doctrine (v3.3)

Each Platform CES may spawn single-task, temporary sub-nodes to process narrow data streams. Each sub-node holds only its slice of context, reports clean summaries upward, and prevents context poisoning at the CES level.

  • Sub-nodes run on cheaper model tiers (Haiku, Flash, GPT-4o-mini) to preserve budget.
  • Sub-nodes NEVER communicate laterally — all coordination flows up to the Platform CES.
  • Sub-nodes are ephemeral: spawned for a task, terminated when summary is delivered.
  • The swarm is FRACTAL: a sub-node may spawn its own sub-nodes if its scope subdivides further. Cost, not architecture, is the limit.
